# CCNA Enterprise Campus Lab — VLANs, HSRP, EtherChannel & OSPF

A Cisco Packet Tracer lab built to practice core CCNA-level switching and routing concepts on a
two-tier (access/distribution) campus network with dual core routers and dual ISP uplinks.

## Overview

This lab simulates a small company's head-office network in Cisco Packet Tracer. The company has
five departments (Native/general, Admin, HR, IT, Sales), each on its own VLAN, connected to the
internet through two independent ISP links so the network keeps working even if a switch, a
router, or one internet connection fails.

The idea was to build one lab that touches most of the switching and routing topics on the CCNA
exam, wired together the way they'd actually be used in a real network rather than as isolated
exercises:

- **Access layer** (A-S-1, A-S-2) — where end users plug in. Each port belongs to one VLAN, and
  switches uplink to distribution using trunks.
- **Distribution layer** (D-ML-S1, D-ML-S2) — where VLANs are routed between each other
  (inter-VLAN routing) and where the default gateway for every VLAN is made redundant with HSRP.
- **Core layer** (R-Core1, R-Core2) — pure routers, running OSPF, each with its own link out to
  the ISP.
- **ISP** — a router standing in for "the internet," reachable from both core routers.

Rather than just getting the network functional, the goal was to remove every single point of
failure between an end-user PC and the internet:
- If one **access-to-distribution link** fails, the other trunk still gets traffic through.
- If one **distribution switch** fails, HSRP fails the default gateway over to the other one.
- If one **core router** fails, OSPF reroutes through the other.
- If one **ISP link** fails, the floating static route on R-Core2 takes over.

Everything below documents how each piece was built and why, plus the addressing plan and a few
real mistakes made (and fixed) along the way.

## Objectives

- [x] VLAN creation and access port assignment
- [x] SVI (Switched Virtual Interface) configuration on distribution switches
- [x] HSRP (Hot Standby Router Protocol) implementation for first-hop redundancy
- [x] Rapid PVST+ configuration with PortFast + BPDU Guard on edge ports
- [x] EtherChannel (LACP) between distribution switches with src-dst-ip load balancing
- [x] Single-area OSPF configuration with backup links and a standby router
- [x] Static floating route configuration
- [x] VLSM (Variable Length Subnet Masking)
- [x] Structured IP addressing plan

## Topology Overview

| Layer            | Devices            | Role |
|-------------------|---------------------|------|
| Access            | A-S-1, A-S-2        | Layer 2 access switches — end-user VLAN ports, trunk uplinks to both distribution switches |
| Distribution      | D-ML-S1, D-ML-S2    | Multilayer switches — VLAN SVIs/HSRP, STP root, EtherChannel to each other, OSPF to core |
| Core              | R-Core1, R-Core2    | Routers — OSPF backbone, redundant paths to ISP |
| Edge/ISP          | ISP                 | Simulated internet edge, OSPF peer, Loopback0 (8.8.8.8) as a "internet" test target |
| End devices       | PC1–PC6             | One host per VLAN (PC6 sits on the Native VLAN behind A-S-2 to test end-to-end reachability from both access switches) |

**Redundancy design:**
- Every access switch trunks to **both** distribution switches (dual uplinks).
- The two distribution switches are connected via a **Port-Channel (LACP, 2×GigabitEthernet)**.
- Each distribution switch has **two independent links to two different core routers**, so no single core router or single link is a point of failure.
- Each core router has its **own link to the ISP router**, giving two independent paths to the "internet."

## IP Addressing Plan

### VLANs / User Subnets

| VLAN | Name   | Subnet             | D-ML-S1 SVI     | D-ML-S2 SVI     | HSRP VIP        | Active HSRP Router |
|------|--------|---------------------|------------------|------------------|-----------------|---------------------|
| 10   | Native | 192.168.10.0/24     | .253             | .252             | .254            | D-ML-S1 (priority 110) |
| 20   | Admin  | 192.168.20.0/24     | .253             | .252             | .254            | D-ML-S1 (priority 120) |
| 30   | HR     | 192.168.30.0/24     | .253             | .252             | .254            | D-ML-S1 (priority 130) |
| 40   | IT     | 192.168.40.0/24     | .252             | .253             | .254            | D-ML-S2 (priority 140) |
| 50   | Sales  | 192.168.50.0/24     | .252             | .253             | .254            | D-ML-S2 (priority 150) |

STP root roles were aligned with HSRP active roles so the active gateway for a VLAN is also the
STP root for that VLAN, avoiding suboptimal traffic paths:
- **D-ML-S1** — STP root primary for VLANs 10, 20, 30 / root secondary for VLANs 40, 50
- **D-ML-S2** — STP root primary for VLANs 40, 50 / root secondary for VLANs 10, 20, 30

ISP Loopback0: `8.8.8.8/32` — used as a simulated internet destination for end-to-end reachability tests.

## Access Layer Configuration

Each access switch (A-S-1, A-S-2) is configured with:
- One VLAN per department, access ports assigned per PC
- `spanning-tree portfast` + `spanning-tree bpduguard enable` on every user-facing port
- Trunk uplinks to **both** distribution switches carrying all relevant VLANs, with VLAN 10 as native
- Rapid PVST+ (`spanning-tree mode rapid-pvst`)
- Unused ports shut down and labeled `#Not in use#`

## Distribution Layer Configuration

Each distribution switch (D-ML-S1, D-ML-S2) is configured with:
- All five VLANs and matching SVIs
- **HSRP** on every SVI, with priority tuned per VLAN and `preempt` enabled so the intended router reclaims the active role after recovering from a failure
- **LACP EtherChannel** (`channel-group mode active`) bundling two links into `Port-channel1`, routed (`no switchport`) as a point-to-point Layer 3 link between the two distribution switches, with `port-channel load-balance src-dst-ip`
- `ip routing` enabled (inter-VLAN routing at the distribution layer)
- Dual routed uplinks to the two core routers
- **OSPF area 0** with the access-switch-facing trunk ports set as `passive-interface` (no OSPF neighbors expected there)

## Core Layer Configuration

- **R-Core1** and **R-Core2** each run OSPF area 0 toward the distribution switches and advertise a default route into OSPF via `default-information originate`, giving the internal network a path to the simulated internet.
- **R-Core2**'s static default route to the ISP is configured as a **floating static route** (administrative distance 200), so it is only used if R-Core1's path is unavailable.
- **R-Core2** additionally raises `ip ospf cost` on its links toward the distribution switches, biasing internal traffic toward the R-Core1 path under normal conditions while keeping R-Core2 as a live backup.
- **ISP** peers with both core routers over OSPF and represents the outside network.

## Troubleshooting Notes / Lessons Learned

Kept here intentionally, since working through these was as valuable as the working config:

- On **D-ML-S1**, VLAN 30's HSRP group was left without `preempt`, and VLANs 40/50 were left at
  default HSRP priority — worth revisiting so failover/failback behavior is fully symmetric with
  D-ML-S2.
- While configuring VLAN 50 on D-ML-S1, the SVI address was first mistakenly typed as
  `192.168.40.252` (a copy/paste slip from the VLAN 40 SVI) before being corrected to
  `192.168.50.252`.
- The ISP router's Loopback0 (`8.8.8.8`) is **not** included in an OSPF `network` statement, so it
  is not currently advertised — a good next step would be adding it to area 0 (or advertising it
  as a static/default route) so it's reachable for `ping 8.8.8.8` tests from inside the network.
- A few command abbreviations (`sw m tr`, `spann portfast`, etc.) were used during configuration —
  fully expanded, these are `switchport mode trunk` and `spanning-tree portfast`.
